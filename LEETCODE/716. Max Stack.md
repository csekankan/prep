# 716. Max Stack

## Problem Statement
https://algomaster.io/practice/dsa/max-stack?list=am-600
Design a max stack data structure that supports the stack operations and supports finding the stack's maximum element.

Implement the `MaxStack` class:

- `MaxStack()` — Initializes the stack object.
- `void push(int x)` — Pushes element `x` onto the stack.
- `int pop()` — Removes the element on top of the stack and returns it.
- `int top()` — Gets the element on the top of the stack without removing it.
- `int peekMax()` — Retrieves the maximum element in the stack without removing it.
- `int popMax()` — Retrieves the maximum element in the stack and removes it. If there is more than one maximum element, only remove the **top-most** one.

### Example

```
Input:  ["MaxStack","push","push","push","top","popMax","top","peekMax","pop","top"]
        [[],[5],[1],[5],[],[],[],[],[],[]]
Output: [null,null,null,null,5,5,1,5,1,5]

MaxStack stk = new MaxStack();
stk.push(5);   // [5]           top=5, max=5
stk.push(1);   // [5, 1]        top=1, max=5
stk.push(5);   // [5, 1, 5]     top=5, max=5
stk.top();     // return 5
stk.popMax();  // return 5       → [5, 1]
stk.top();     // return 1
stk.peekMax(); // return 5
stk.pop();     // return 1       → [5]
stk.top();     // return 5
```

### Constraints

- `-10^7 <= x <= 10^7`
- At most `10^5` calls will be made to `push`, `pop`, `top`, `peekMax`, and `popMax`.
- There will be at least one element in the stack when `pop`, `top`, `peekMax`, or `popMax` is called.

---

## Understanding the Problem

We need to build a stack that does everything a normal stack does (`push`, `pop`, `top`) but also supports two extra operations: finding the maximum element and removing the maximum element.

The tricky part is `popMax`. Regular stacks only let you remove from the top, but here we need to reach into the middle of the stack and pull out the maximum — and if there are duplicates, we remove the one closest to the top.

**Think about it this way.** A normal stack gives you O(1) push, pop, and top. Adding `peekMax` isn't too bad since you could track the running maximum with a second stack (that's the classic Min Stack trick). But `popMax` changes everything. The maximum might be buried deep in the stack. When you remove it, the elements above it need to stay in order, and the "new maximum" might be different. This tension between **stack ordering** and **max-tracking** is what makes this problem hard.

> The key question is: can we find a way to support all five operations efficiently, or do we have to accept that some operations will be slow?

### Key Constraints

- `-10^7 <= x <= 10^7` — Values can be negative and large. We need to handle the full integer range. No assumptions about values being positive or bounded to a small range.
- **At most 10^5 calls** — With 10^5 operations, an O(n) per operation approach gives us 10^10 worst case, which is too slow. We need O(log n) per operation for the optimal solution, though O(n) per `popMax` with O(1) for everything else can sometimes squeeze through.

---

## Approach 1: Two Stacks (Brute Force popMax)

### Intuition

The simplest way to think about this: use two stacks. The main stack holds all elements in order. The second stack (the "max stack") tracks the running maximum at each level, just like the classic Min Stack problem.

For push, pop, top, and peekMax, we get O(1). The second stack always has the current maximum at its top because every time we push a value, we also push `max(value, currentMax)` onto the max stack.

But `popMax` is where things get messy. The maximum might not be at the top of the main stack. So we need to pop elements off the top one by one until we find the maximum, remove it, then push everything back. During this rebuild, we also need to rebuild the max stack so it stays correct.

It's not pretty, but it's correct and easy to implement. For most interview scenarios where n is small, this works fine.

### Algorithm

1. Maintain two stacks: `stack` (the main stack) and `maxStack` (tracks the running maximum).
2. `push(x)`: Push `x` onto `stack`. Push `max(x, maxStack.peek())` onto `maxStack` (or just `x` if `maxStack` is empty).
3. `pop()`: Pop from both `stack` and `maxStack`. Return the value from `stack`.
4. `top()`: Return `stack.peek()`.
5. `peekMax()`: Return `maxStack.peek()`.
6. `popMax()`: Get the maximum from `maxStack.peek()`. Pop elements into a temp stack until you find the max. Remove it. Push everything back using `push()` to rebuild the max stack.

### Code

```python
class MaxStack:
    def __init__(self):
        self.stack = []
        self.max_stack = []

    def push(self, x: int) -> None:
        self.stack.append(x)
        if self.max_stack:
            self.max_stack.append(max(x, self.max_stack[-1]))
        else:
            self.max_stack.append(x)

    def pop(self) -> int:
        self.max_stack.pop()
        return self.stack.pop()

    def top(self) -> int:
        return self.stack[-1]

    def peekMax(self) -> int:
        return self.max_stack[-1]

    def popMax(self) -> int:
        max_val = self.max_stack[-1]
        temp = []
        while self.stack[-1] != max_val:
            temp.append(self.pop())
        self.pop()  # remove the max element
        while temp:
            self.push(temp.pop())
        return max_val
```

### Complexity

| Operation | Time | Space |
|-----------|------|-------|
| push      | O(1) | O(n)  |
| pop       | O(1) |       |
| top       | O(1) |       |
| peekMax   | O(1) |       |
| popMax    | **O(n)** |   |

The bottleneck is `popMax`. Every call might pop the entire stack to find the max. What if we used a heap to find the max in O(log n) and avoided the linear scan?

## Approach 2: Heap + Lazy Deletion

### Intuition

We already know how to find the maximum quickly — just use a max-heap. And we know how to maintain stack order — just use a regular stack (or list). The challenge is keeping these two structures in sync when we do `popMax`.

The trick is **lazy deletion**. Instead of physically removing an element from both structures at the same time, we just mark it as "deleted" and clean it up later when we encounter it. Each element gets a **unique ID** (a counter that increments with each push), so we can tell which specific element to delete even when there are duplicate values.

When we call `popMax`, we pop from the heap to get the maximum, record its ID as deleted, but don't touch the stack yet. When we call `pop` or `top`, we check if the top of the stack has been marked as deleted, and if so, we skip over it.

### Algorithm

1. Maintain a stack (list of `(value, id)` pairs), a max-heap of `(-value, -id)` pairs, a set of deleted IDs, and a counter for generating unique IDs.
2. `push(x)`: Assign a new unique ID. Push `(x, id)` onto the stack and `(-x, -id)` onto the heap. Increment the counter.
3. `pop()`: Skip any top-of-stack entries whose IDs are in the deleted set. Pop the top entry, add its ID to the deleted set, and return the value.
4. `top()`: Skip any top-of-stack entries whose IDs are in the deleted set. Return the top value.
5. `peekMax()`: Skip any top-of-heap entries whose IDs are in the deleted set. Return the top value.
6. `popMax()`: Skip any top-of-heap entries whose IDs are in the deleted set. Pop the top entry, add its ID to the deleted set, and return the value.

### Why a unique ID matters

Without IDs, duplicate values become ambiguous. If we push `5, 1, 5` and call `popMax`, we want to remove the **top-most** `5` (index 2). With IDs, the heap stores `(-5, -2)` and `(-5, -0)`. Since Python compares tuples left-to-right, `(-5, -2)` comes first — that's the most recent `5`. The ID breaks ties correctly.

### Code

```python
import heapq

class MaxStack:
    def __init__(self):
        self.stack = []
        self.heap = []
        self.deleted = set()
        self.counter = 0

    def push(self, x: int) -> None:
        self.stack.append((x, self.counter))
        heapq.heappush(self.heap, (-x, -self.counter))
        self.counter += 1

    def pop(self) -> int:
        while self.stack and self.stack[-1][1] in self.deleted:
            self.deleted.discard(self.stack[-1][1])
            self.stack.pop()
        val, idx = self.stack.pop()
        self.deleted.add(idx)
        return val

    def top(self) -> int:
        while self.stack and self.stack[-1][1] in self.deleted:
            self.deleted.discard(self.stack[-1][1])
            self.stack.pop()
        return self.stack[-1][0]

    def peekMax(self) -> int:
        while self.heap and -self.heap[0][1] in self.deleted:
            self.deleted.discard(-self.heap[0][1])
            heapq.heappop(self.heap)
        return -self.heap[0][0]

    def popMax(self) -> int:
        while self.heap and -self.heap[0][1] in self.deleted:
            self.deleted.discard(-self.heap[0][1])
            heapq.heappop(self.heap)
        val, idx = heapq.heappop(self.heap)
        self.deleted.add(-idx)
        return -val
```

### Complexity

| Operation | Time (amortized) | Space |
|-----------|-----------------|-------|
| push      | O(log n)        | O(n)  |
| pop       | O(log n)*       |       |
| top       | O(log n)*       |       |
| peekMax   | O(log n)*       |       |
| popMax    | O(log n)*       |       |

\* Amortized — each element is lazily cleaned up at most once, but a single call can be slow in pathological cases.

## Approach 3: TreeMap + Doubly Linked List (Optimal)

### Intuition

Let's think about what each operation truly needs:

- **Push and pop** need stack ordering (know what's on top).
- **PeekMax and popMax** need fast access to the maximum.
- **PopMax** needs to remove an element from the **middle** of the stack without disturbing the order.

No single data structure gives us all of this. But two data structures working together can:

- A **doubly linked list** maintains stack order and lets us remove any node in O(1) if we have a pointer to it.
- A **balanced BST** (TreeMap / SortedDict) lets us find and access the maximum in O(log n).

The bridge between them: the TreeMap maps each value to a list of linked list node references. When we push a value, we add a node to the tail of the linked list and also record a reference to that node in the TreeMap. When we call `popMax`, we look up the maximum key in the TreeMap, grab the most recently pushed node with that value, remove it from the linked list in O(1), and remove the reference from the TreeMap. **No lazy deletion needed.**

### Why a doubly linked list instead of a Python list?

A Python list is backed by an array. Removing from the middle is O(n) because everything after the removed element shifts. A doubly linked list removes any node in O(1) — just update the prev/next pointers. This is the key insight that makes the optimal solution work.

### Why dummy head and tail nodes?

Two dummy (sentinel) nodes eliminate edge-case checks for empty list or single element. Without them, every `_add` and `_remove` would need `if node.prev` / `if node.next` checks. With dummies, real nodes are always sandwiched between `head` and `tail`, so `node.prev` and `node.next` are never `None`.

### Algorithm

1. Maintain a doubly linked list where the tail represents the top of the stack.
2. Maintain a TreeMap (SortedDict) where each key is a value, and the associated data is a list of node references (in push order).
3. `push(x)`: Create a new node, append it to the tail of the linked list. In the TreeMap, add this node reference to the list under key `x`.
4. `pop()`: Remove the tail node from the linked list. Remove the last node reference from the TreeMap under the node's value. If that list is empty, remove the key.
5. `top()`: Return the tail node's value.
6. `peekMax()`: Return the last key in the TreeMap (the maximum).
7. `popMax()`: Get the last key from the TreeMap. Get the last node reference from that key's list. Remove that node from the doubly linked list. Remove the reference from the TreeMap.

### Code

```python
from sortedcontainers import SortedList

class Node:
    __slots__ = ('val', 'idx', 'prev', 'next')

    def __init__(self, val, idx):
        self.val = val
        self.idx = idx
        self.prev = None
        self.next = None

class MaxStack:
    def __init__(self):
        self.head = Node(0, 0)
        self.tail = Node(0, 0)
        self.head.next = self.tail
        self.tail.prev = self.head
        self.sorted = SortedList(key=lambda node: (node.val, node.idx))
        self.counter = 0

    def _add_to_tail(self, node: Node) -> None:
        prev = self.tail.prev
        prev.next = node
        node.prev = prev
        node.next = self.tail
        self.tail.prev = node

    def _remove_node(self, node: Node) -> None:
        node.prev.next = node.next
        node.next.prev = node.prev

    def push(self, x: int) -> None:
        node = Node(x, self.counter)
        self.counter += 1
        self._add_to_tail(node)
        self.sorted.add(node)

    def pop(self) -> int:
        node = self.tail.prev
        self._remove_node(node)
        self.sorted.remove(node)
        return node.val

    def top(self) -> int:
        return self.tail.prev.val

    def peekMax(self) -> int:
        return self.sorted[-1].val

    def popMax(self) -> int:
        node = self.sorted[-1]
        self._remove_node(node)
        self.sorted.remove(node)
        return node.val
```

### Complexity

| Operation | Time      | Space |
|-----------|-----------|-------|
| push      | O(log n)  | O(n)  |
| pop       | O(log n)  |       |
| top       | **O(1)**  |       |
| peekMax   | **O(1)**  |       |
| popMax    | O(log n)  |       |

Every operation is O(log n) worst-case (not amortized). `top()` and `peekMax()` are O(1) since we just peek at the DLL tail or SortedList end.

## Common Pitfalls

### 1. Index fragility with lazy deletion
When using positional indices (like `len(stack)-1`) as identifiers, they become stale as elements are lazily deleted. A globally incrementing counter is far more robust — each element gets a permanent, unique ID that never changes.

### 2. Off-by-one after pop
After `stack.pop()`, `len(stack)` decreases by 1. Code like `deleatedIndex.remove(len(stack))` vs `deleatedIndex.remove(len(stack)-1)` behaves differently depending on whether it runs before or after the pop. Using unique IDs instead of positional indices avoids this class of bugs entirely.

### 3. Inconsistent cleanup across operations
If `peekMax` has a loop to skip stale heap entries but `popMax` doesn't, they'll disagree on what the current max is. Every operation that touches a lazy-deletion structure must have the same cleanup logic.

### 4. Forgetting to handle duplicates
`popMax` must remove the **top-most** occurrence of the maximum. Using `(-value, -id)` tuples in the heap ensures the most recently pushed duplicate sorts first.

### 5. Not cleaning up the deleted set
If you skip deleted entries (e.g., popping past them in the stack) but never remove them from the deleted set, the set grows unboundedly and stale markers can cause subtle bugs.

## My Final Solution (Heap + Lazy Deletion — Custom Variant)

```python
import heapq
from collections import defaultdict

class MaxStack:
    def __init__(self):
        self.maxTracker = []
        self.values = defaultdict(list)
        self.stack = []
        self.deleatedIndex = set()

    def push(self, x: int) -> None:
        self.stack.append(x)
        self.values[x].append(len(self.stack) - 1)
        heapq.heappush(self.maxTracker, -1 * x)

    def pop(self) -> int:
        while self.stack and len(self.stack) - 1 in self.deleatedIndex:
            self.deleatedIndex.remove(len(self.stack) - 1)
            self.stack.pop()
        val = self.stack.pop()
        self.values[val].pop()
        if len(self.values[val]) == 0:
            del self.values[val]
        return val

    def top(self) -> int:
        while self.stack and len(self.stack) - 1 in self.deleatedIndex:
            self.stack.pop()
            self.deleatedIndex.remove(len(self.stack))
        return self.stack[-1]

    def peekMax(self) -> int:
        while self.maxTracker:
            curMax = self.maxTracker[0]
            if -1 * curMax not in self.values:
                heapq.heappop(self.maxTracker)
            else:
                return -1 * curMax
        return -1

    def popMax(self) -> int:
        while self.maxTracker:
            curMax = self.maxTracker[0]
            if -1 * curMax not in self.values:
                heapq.heappop(self.maxTracker)
            else:
                break
        curMax = self.maxTracker[0]
        val = -1 * curMax
        if val in self.values:
            index = self.values[val].pop()
            self.deleatedIndex.add(index)
            if len(self.values[val]) == 0:
                del self.values[val]
        return -1 * heapq.heappop(self.maxTracker)
```

## Gaps in My Solution vs Clean Heap + Lazy Deletion

| Gap | My solution | Clean version |
|-----|-------------|---------------|
| **Element identity** | Uses positional index `len(stack)-1` which shifts as elements are lazily deleted — caused multiple off-by-one bugs | Uses a monotonic `counter` as a permanent unique ID that never changes |
| **What's stored** | Stack stores raw values, a separate `values` dict maps value → list of indices | Stack stores `(value, id)` tuples — each entry is self-describing |
| **Heap entries** | Stores just `-value`, so duplicates are indistinguishable in the heap | Stores `(-value, -id)` tuples — duplicates are distinguishable, and the most recent one sorts first |
| **Staleness check** | Checks `if value not in self.values` — works only because we delete the key when count hits 0, but it conflates "value was fully removed" with "value still exists somewhere" | Checks `if id in self.deleted` — precise per-element check, no ambiguity |
| **Cleanup symmetry** | `pop()`/`top()` clean the stack, `peekMax()`/`popMax()` clean the heap — had to manually duplicate the cleanup loop (forgot it initially in `popMax`) | Same pattern everywhere: "skip deleted entries, then act" — symmetric and hard to get wrong |
| **Off-by-one risk** | `deleatedIndex.remove(len(stack)-1)` vs `remove(len(stack))` depends on before/after pop — this was the #1 recurring bug | No indices at all, just IDs — off-by-one is structurally impossible |
| **`popMax` tie-breaking** | `values[val].pop()` pops the last index, which is the most recent push — correct, but only because the list is append-ordered. Not enforced by the data structure | `(-value, -id)` heap ordering *guarantees* the most recent duplicate is popped first — enforced structurally |

### Key Takeaway

> The core gap is **using positional indices as identity**. Positions change as elements are removed, making every operation that touches `deleatedIndex` fragile. A monotonic counter gives each element a **permanent name** — this single change eliminates the entire class of off-by-one bugs.

## Better Version of My Approach (Clean Heap + Lazy Deletion)

Same strategy as my solution (heap + lazy deletion), but with unique IDs instead of positional indices:

```python
import heapq

class MaxStack:
    def __init__(self):
        self.stack = []
        self.heap = []
        self.deleted = set()
        self.counter = 0

    def push(self, x: int) -> None:
        self.stack.append((x, self.counter))
        heapq.heappush(self.heap, (-x, -self.counter))
        self.counter += 1

    def _clean_stack(self):
        while self.stack and self.stack[-1][1] in self.deleted:
            self.deleted.discard(self.stack[-1][1])
            self.stack.pop()

    def _clean_heap(self):
        while self.heap and -self.heap[0][1] in self.deleted:
            self.deleted.discard(-self.heap[0][1])
            heapq.heappop(self.heap)

    def pop(self) -> int:
        self._clean_stack()
        val, idx = self.stack.pop()
        self.deleted.add(idx)
        return val

    def top(self) -> int:
        self._clean_stack()
        return self.stack[-1][0]

    def peekMax(self) -> int:
        self._clean_heap()
        return -self.heap[0][0]

    def popMax(self) -> int:
        self._clean_heap()
        val, idx = heapq.heappop(self.heap)
        self.deleted.add(-idx)
        return -val
```

**What changed from my version:**
1. **`counter` instead of `len(stack)-1`** — permanent IDs, no off-by-one possible
2. **`(value, id)` tuples** — no separate `values` dict needed
3. **`_clean_stack` / `_clean_heap` helpers** — cleanup logic written once, used everywhere (no risk of forgetting it in one method)
4. **`(-value, -id)` in heap** — ties broken by most-recent-first automatically

## Quick Revision Checklist

- [ ] **Two Stacks** — O(1) everything except O(n) popMax. Use when n is small or in an interview if you need a quick correct solution.
- [ ] **Heap + Lazy Deletion** — O(log n) amortized. Use unique IDs, not positional indices. Store `(value, id)` pairs. Extract cleanup into helpers. Remember `(-value, -id)` for correct tie-breaking.
- [ ] **DLL + SortedList** — O(log n) worst-case. Doubly linked list for O(1) mid-removal. SortedList/TreeMap for O(log n) max access. Dummy head/tail to avoid edge cases. This is the optimal solution.
- [ ] **Core insight**: The tension is between stack ordering and max-tracking. No single structure handles both — you need two structures bridged by node references or unique IDs.
- [ ] **Lazy deletion pitfall**: Every operation that touches a lazy structure must have the same cleanup logic. If `peekMax` cleans but `popMax` doesn't, they disagree.
- [ ] **Index vs ID**: Never use mutable positional indices as element identity in lazy deletion. Use a monotonic counter.
