# 1846. Maximum Element After Decrementing and Rearranging

## Problem

Given an array `nums`, you can:
1. **Rearrange** elements in any order
2. **Decrement** any element (but never below 1)

**Goal:** Maximize the maximum element in the array.

**Key constraint:** After sorting optimally, each element can differ from the previous by at most 1.

---

## Intuition

After rearranging, the optimal array looks like a consecutive sequence starting from 1:
```
[1, 2, 3, 4, ..., k]
```
Each element is at most 1 more than the previous. We want to find the largest `k` achievable.

**Why can't we do better?**
- `nums[0]` must be 1 (we can only decrement, so set the smallest slot to 1)
- Each next slot can increase by at most 1
- So the max possible answer with `n` elements is `n`

---

## Approach 1 — Sort + Greedy (O(n log n))

**Idea:** Sort the array, force `nums[0] = 1`, then for each subsequent element cap it at `prev + 1`.

```python
def maximumElementAfterDecrementingAndRearranging(nums):
    nums.sort()
    nums[0] = 1
    for i in range(1, len(nums)):
        nums[i] = min(nums[i], nums[i-1] + 1)
    return nums[-1]
```

**Trace:** `nums = [1, 3, 3, 4]`
```
sort  →  [1, 3, 3, 4]
i=1   →  min(3, 1+1) = 2   →  [1, 2, 3, 4]
i=2   →  min(3, 2+1) = 3   →  [1, 2, 3, 4]
i=3   →  min(4, 3+1) = 4   →  [1, 2, 3, 4]
return 4  ✓
```

**Why `min(nums[i], nums[i-1] + 1)`?**
- `nums[i-1] + 1` is the max value this slot can hold (can't jump by more than 1)
- `nums[i]` is the original value (can't increase, only decrease)
- Take the smaller of the two

## Approach 2 — Frequency Bucket (O(n)) ← Optimal

**Key observations:**
1. Answer ≤ `n` (n elements, each differs by at most 1, starting from 1)
2. Any element > n behaves the same as n → cap all values at n
3. Build a frequency array, then do one left-to-right sweep

**Core formula:**
```
cur = min(cur + freq[i], i)
```

- `cur` = "I can guarantee positions 1 through cur are filled"
- `freq[i]` new elements at bucket i → extend reach by `freq[i]`
- Cap at `i` because elements with value i **cannot be placed above position i** (no increment allowed)

```python
def maximumElementAfterDecrementingAndRearranging(nums):
    n = len(nums)
    freq = [0] * (n + 1)

    for num in nums:
        freq[min(num, n)] += 1      # cap at n

    cur = 0
    for i in range(1, n + 1):
        cur = min(cur + freq[i], i)

    return cur
```

**Trace:** `nums = [1, 3, 3, 4]`, n=4
```
freq = [0, 1, 0, 2, 1]
        0  1  2  3  4

i=1: cur = min(0+1, 1) = 1
i=2: cur = min(1+0, 2) = 1   ← no elements at bucket 2, cur stays at 1
i=3: cur = min(1+2, 3) = 3   ← 2 elements at bucket 3 jump cur from 1 to 3
i=4: cur = min(3+1, 4) = 4   ← 1 element at bucket 4 extends to 4
return 4  ✓
```

**Trace:** `nums = [2, 2, 2, 2, 2]`, n=5
```
freq = [0, 0, 5, 0, 0, 0]

i=1: cur = min(0+0, 1) = 0
i=2: cur = min(0+5, 2) = 2   ← 5 elements but capped at 2 (can't go above bucket value)
i=3: cur = min(2+0, 3) = 2
i=4: cur = min(2+0, 4) = 2
i=5: cur = min(2+0, 5) = 2
return 2  ✓
```

## My Original Approach — What Was Wrong

### The Code

```python
# Frequency array built correctly
a = [0] * (n + 1)
for num in nums:
    if num >= n: a[n] += 1
    else: a[num] += 1

# Greedy sweep with two pointers
r = n
l = 1
ans = 0
extraCount = 0

while l <= r:
    while r >= l and a[r] <= 0:
        r -= 1
    if a[l] == 0:
        if r > l and a[r] > 0:
            ans += 1
            a[r] -= 1          # steal from highest bucket
        else:
            extraCount -= 1    # try to use surplus
    else:
        extraCount += (a[l] - 1)
        ans += 1
    l += 1
```

### Bug 1 — Stealing from the Highest Bucket

When position `l` is empty, the code steals from `r` (the highest available bucket).

**Why this is wrong:** It uses a high-value element for a low position, leaving the high position unfilled later.

**Failing case:** `nums = [1, 3, 3, 4]`, n=4
```
a = [0, 1, 0, 2, 1]

l=1: a[1]=1 → ans=1, extraCount=0
l=2: a[2]=0 → steal from r=4 → ans=2, a[4]=0   ← bucket 4 wasted at position 2!
l=3: a[3]=2 → ans=3, extraCount=1
l=4: a[4]=0 → r=3, not > l=4 → extraCount -= 1 = 0  ← position 4 can't be filled

returns 3  ✗   expected 4
```

The element at bucket 4 (value 4) should have stayed at position 4. The surplus from bucket 3 should have been decremented to fill position 2.

**What should happen:** Prefer stealing from the **nearest available bucket above l**, not the farthest. But even this is hard to implement correctly because you don't know future surpluses.

### Bug 2 — Surplus Cannot Fill Higher Positions

```python
else:
    extraCount -= 1    # trying to use surplus for position l
```

`extraCount` holds surplus elements from buckets **below** l. An element from bucket 2 has value 2. It **cannot** fill position 4 — that would require incrementing it from 2 to 4, which is not allowed.

**So `extraCount` is conceptually useless for filling higher positions.** Surplus elements are simply wasted (discarded).

The `cur = min(cur + freq[i], i)` formula handles this naturally — the `min` discards surplus silently.

### Bug 3 — Missing `ans += 1` (Secondary)

Even if we were to use extraCount (which we can't), the code decrements it without incrementing `ans`:
```python
extraCount -= 1    # ← fills position but doesn't count it!
```

### Bug 4 — Unreachable Code

```python
    return ans        # code returns here

return minVal         # this line never runs
```

## The Core Mental Model Gap

| My Thinking | Correct Thinking |
|---|---|
| "Steal from highest bucket to fill empty spots" | "Elements can only go DOWN to fill lower positions, not up" |
| "Surplus can be carried forward to fill higher positions" | "Surplus is wasted — elements can't be incremented" |
| Need: two pointers + extra state | Need: one variable `cur`, forward sweep only |
| O(n) but with hidden correctness bugs | O(n) clean and provably correct |

**The one rule that the two-pointer approach violates:**
> An element at bucket `i` can fill any position ≤ i (by decrementing), but **never** a position > i (that would require incrementing).

The `cur = min(cur + freq[i], i)` formula encodes this rule directly in the `min(..., i)` cap.
