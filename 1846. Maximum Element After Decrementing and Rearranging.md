# 1846. Maximum Element After Decrementing and Rearranging

## Problem

Given an array `nums`, you can:
1. **Rearrange** elements in any order
2. **Decrement** any element (but never below 1)

**Goal:** Maximize the maximum element in the array.

**Key constraint:** After sorting optimally, each element can differ from the previous by at most 1.

---

## Intuition

After rearranging, the optimal array looks like a consecutive sequence starting from 1:
```
[1, 2, 3, 4, ..., k]
```
Each element is at most 1 more than the previous. We want to find the largest `k` achievable.

**Why can't we do better?**
- `nums[0]` must be 1 (we can only decrement, so set the smallest slot to 1)
- Each next slot can increase by at most 1
- So the max possible answer with `n` elements is `n`

---

## Approach 1 — Sort + Greedy (O(n log n))

**Idea:** Sort the array, force `nums[0] = 1`, then for each subsequent element cap it at `prev + 1`.

```python
def maximumElementAfterDecrementingAndRearranging(nums):
    nums.sort()
    nums[0] = 1
    for i in range(1, len(nums)):
        nums[i] = min(nums[i], nums[i-1] + 1)
    return nums[-1]
```

**Trace:** `nums = [1, 3, 3, 4]`
```
sort  →  [1, 3, 3, 4]
i=1   →  min(3, 1+1) = 2   →  [1, 2, 3, 4]
i=2   →  min(3, 2+1) = 3   →  [1, 2, 3, 4]
i=3   →  min(4, 3+1) = 4   →  [1, 2, 3, 4]
return 4  ✓
```

**Why `min(nums[i], nums[i-1] + 1)`?**
- `nums[i-1] + 1` is the max value this slot can hold (can't jump by more than 1)
- `nums[i]` is the original value (can't increase, only decrease)
- Take the smaller of the two

## Approach 2 — Frequency Bucket (O(n)) ← Optimal

**Key observations:**
1. Answer ≤ `n` (n elements, each differs by at most 1, starting from 1)
2. Any element > n behaves the same as n → cap all values at n
3. Build a frequency array, then do one left-to-right sweep

**Core formula:**
```
cur = min(cur + freq[i], i)
```

- `cur` = "I can guarantee positions 1 through cur are filled"
- `freq[i]` new elements at bucket i → extend reach by `freq[i]`
- Cap at `i` because elements with value i **cannot be placed above position i** (no increment allowed)

```python
def maximumElementAfterDecrementingAndRearranging(nums):
    n = len(nums)
    freq = [0] * (n + 1)

    for num in nums:
        freq[min(num, n)] += 1      # cap at n

    cur = 0
    for i in range(1, n + 1):
        cur = min(cur + freq[i], i)

    return cur
```

**Trace:** `nums = [1, 3, 3, 4]`, n=4
```
freq = [0, 1, 0, 2, 1]
        0  1  2  3  4

i=1: cur = min(0+1, 1) = 1
i=2: cur = min(1+0, 2) = 1   ← no elements at bucket 2, cur stays at 1
i=3: cur = min(1+2, 3) = 3   ← 2 elements at bucket 3 jump cur from 1 to 3
i=4: cur = min(3+1, 4) = 4   ← 1 element at bucket 4 extends to 4
return 4  ✓
```

**Trace:** `nums = [2, 2, 2, 2, 2]`, n=5
```
freq = [0, 0, 5, 0, 0, 0]

i=1: cur = min(0+0, 1) = 0
i=2: cur = min(0+5, 2) = 2   ← 5 elements but capped at 2 (can't go above bucket value)
i=3: cur = min(2+0, 3) = 2
i=4: cur = min(2+0, 4) = 2
i=5: cur = min(2+0, 5) = 2
return 2  ✓
```

## My Original Approach — What Was Wrong

### The Code

```python
# Frequency array built correctly
a = [0] * (n + 1)
for num in nums:
    if num >= n: a[n] += 1
    else: a[num] += 1

# Greedy sweep with two pointers
r = n
l = 1
ans = 0
extraCount = 0

while l <= r:
    while r >= l and a[r] <= 0:
        r -= 1
    if a[l] == 0:
        if r > l and a[r] > 0:
            ans += 1
            a[r] -= 1          # steal from highest bucket
        else:
            extraCount -= 1    # try to use surplus
    else:
        extraCount += (a[l] - 1)
        ans += 1
    l += 1
```

### ⚠️ THE MISTAKE — Stealing from the Highest Bucket (Root Cause of Everything)

**When position `l` is empty, the code steals from `r` = highest available bucket.**

> **THE BUG: Should steal from the NEAREST available bucket `j` where `l < j ≤ n`, NOT the farthest.**
>
> Stealing from the highest `r` consumes a high-value element at a low position. That high position is now left permanently empty because no element can ever fill it — elements can only be decremented (moved down), never incremented (moved up).

**Failing case:** `nums = [1, 3, 3, 4]`, n=4
```
a = [0, 1, 0, 2, 1]

l=2: a[2]=0 → steal from r=4 (FARTHEST) → ans=2, a[4]=0   ← WRONG: bucket 4 wasted at position 2!
l=4: a[4]=0 → nothing left → position 4 unfilled

returns 3  ✗   expected 4
```

**With nearest instead:**
```
l=2: a[2]=0 → steal from r=3 (NEAREST) → ans=2, a[3]=1   ← bucket 3 element decrements to 2
l=3: a[3]=1 → fill directly → ans=3
l=4: a[4]=1 → fill directly → ans=4  ✓
```

**Why you can't just fix `r` to scan from bottom:**
Even with nearest-first, `r` must scan from `l` upward for each empty position → O(n) inner scan per position → **O(n²) total**. The two-pointer trick of moving `r` from `n` downward only works when stealing from the top is safe, which it isn't here. **There is no clean O(n) two-pointer fix.** The bucket formula sidesteps this entirely.

### Bug 2 — `extraCount` is Meaningless

```python
else:
    extraCount -= 1    # trying to use surplus for position l
```

`extraCount` holds surplus elements from buckets **below** `l`. An element from bucket 2 has value 2 — it **cannot** fill position 4 (that would require incrementing, which is forbidden).

> **Surplus elements are simply wasted. `extraCount` can never help fill a higher position.**

The `cur = min(cur + freq[i], i)` formula handles this naturally — the `min(..., i)` cap silently discards surplus.

### Bug 3 — Unreachable Code

```python
    return ans        # code returns here

return minVal         # this line never runs
```

## The Core Mental Model Gap

| My Thinking | Correct Thinking |
|---|---|
| **"Steal from highest bucket to fill empty spots"** | **"Steal from NEAREST bucket above the gap"** |
| "Surplus can carry forward to fill higher positions" | "Surplus is wasted — elements can't be incremented" |
| Two pointers scanning from both ends | One variable `cur`, forward sweep only |
| O(n) but wrong due to greedy mismatch | O(n) clean and correct |

> **THE ONE-LINE ROOT CAUSE:**
> When position `l` is empty, **always steal from the nearest available bucket above `l`**, not the farthest. Stealing from the farthest permanently destroys an element that was needed for its own high position.
>
> **Why you can't fix the two-pointer to O(n):** To find the nearest bucket above `l`, you must scan upward from `l` — O(n) per position → O(n²) total. The bucket formula `cur = min(cur + freq[i], i)` avoids this scan entirely by accumulating forward.
